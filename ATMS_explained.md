# ATMS (Assumption-based Truth Maintenance System)

## 목적, 해결 과제, 방식, 의의

---

## 1. 탄생 배경: "한 번에 하나의 세계"의 한계

1986년, Xerox PARC의 **Johan de Kleer**는 기존 추론 시스템(JTMS)의 근본적 한계에 직면했습니다.

JTMS(Justification-based Truth Maintenance System)는 각 명제에 **IN 또는 OUT**, 단 하나의 레이블만 부여합니다. 한 시점에 오직 하나의 세계관만 유지할 수 있었습니다.

```
JTMS의 세계: "지금 이 순간, X는 참이다 / 거짓이다"
             → 다른 가능성을 보려면 가정을 철회하고 다시 추론해야 함
```

이것은 **진단(diagnosis)** 같은 문제에서 치명적이었습니다:

> 환자에게 증상 A, B, C가 있다.
> 원인이 질병1일 수도, 질병2일 수도 있다.
>
> JTMS: "질병1이라고 가정" → 추론 → "안 맞네, 철회" → "질병2라고 가정" → 추론...
> 매번 처음부터 다시!

**de Kleer의 질문:**

> "모든 가능한 세계를 동시에 추적할 수는 없을까?"

이 질문의 답이 바로 ATMS입니다.

---

## 2. 해결 과제: 세 가지 핵심 문제

### 2.1 다중 세계 동시 추적

```
JTMS:  노드.label = :IN 또는 :OUT          ← 하나의 답
ATMS:  노드.label = [{A}, {B,C}]           ← "A 세계에서도 참, B+C 세계에서도 참"
```

원본 코드(`atms.lisp:41`)가 이를 명확히 보여줍니다:

```lisp
(label nil)    ; minimal envs believed under   ← 환경들의 리스트!
```

### 2.2 모순 발견 시 자동 격리

JTMS는 모순을 만나면 시스템이 멈추고 사용자가 직접 해결해야 합니다.
ATMS는 모순 환경을 nogood으로 표시하고 나머지 세계는 계속 작동합니다.

```
{A,B}가 모순 → E-5:* {A,B} 로 표시
하지만 {A}, {B}, {C}, {B,C} 세계는 계속 유효!
y=1은 {A,B} 경로가 차단되자 {B,C} 경로로 자동 우회!
```

### 2.3 추론 결과의 재사용

JTMS는 가정을 바꿀 때마다 모든 추론을 처음부터 다시 해야 합니다.
ATMS는 한 번 계산한 추론 결과를 모든 세계에서 공유합니다.

## 3. 방식: 환경(Environment) 기반 레이블 관리

ATMS의 핵심 아이디어:

> **"노드의 참/거짓을 묻지 말고, 어떤 가정 하에서 참인지를 물어라"**

### 3.1 아키텍처 구성 요소

| 구성 요소 | 설명 |
|-----------|------|
| **가정(Assumption)** | 독립적 선택지. 참/거짓을 자유롭게 설정 가능 |
| **환경(Environment)** | 가정들의 집합 = 하나의 "가능한 세계". 예: {A, C} |
| **레이블(Label)** | 최소 환경들의 antichain. 예: [{A}, {B,C}] |
| **Nogood** | 모순인 환경. 예: {A,B}* = "A와 B는 동시에 참일 수 없다" |
| **Justification** | 추론 규칙. antecedents → consequence |

### 3.2 JTMS와의 구조적 비교

| 측면 | JTMS | ATMS |
|------|------|------|
| 노드 레이블 | `:IN` / `:OUT` (이진값) | 환경 리스트 `[{A}, {B,C}]` |
| 동시 세계 | 1개 | **모든 일관된 세계** |
| 모순 처리 | 멈추고 철회 | nogood 표시 후 **자동 격리** |
| 가정 변경 | 철회 → 재추론 | **이미 계산되어 있음** |
| 질문 방식 | "X는 참인가?" | "X는 **어떤 가정 하에서** 참인가?" |
| 핵심 연산 | enable/retract | **weave** (교차곱 + 필터링) |

### 3.3 핵심 함수 파이프라인

```
justify-node(informant, consequence, antecedents)
  ① Justification 구조체 생성
  ② 양방향 링크 등록
  ③ propagate → weave → update
     ├─ 일반 노드: label 갱신 → 하류 재전파
     └─ 모순 노드: new-nogood 5단계 처리
```

### 3.4 new-nogood의 5단계

```
① 금지 도장     → "이 환경은 모순이다"
② label 제거    → "모순 세계에서 참이라는 건 무의미"
③ nogood-table  → "중앙 목록에 등록해서 이후 검사용"
④ 상위 정리     → "큰 금지는 작은 게 커버하니 중복 제거"
⑤ 상향 오염     → "유효한 상위 환경도 전부 즉시 오염"
```

## 4. step-1 예제로 보는 ATMS의 진가

`atest.lisp:43-80` (de Kleer 논문의 예제)

```
초기:  A, B, C (가정)
       J1: A → x=1      x=1.label = [{A}]
       J2: B → y=x      y=x.label = [{B}]
       J3: C → x=z      x=z.label = [{C}]

모순:  nogood({A, B})    → {A,B}* 금지!

시도:  J4: x=1 + y=x → y=1
       {A} × {B} = {A,B} → nogood! 차단!
       y=1.label = []     ← 증명 불가...

전환:  Premise: z=1       z=1.label = [{∅}]  (무조건 참)
       J5: z=1 + x=z → x=1
       {∅} × {C} = {C}
       x=1.label = [{A}, {C}]  ← 두 경로!

재시도: J4 재전파: x=1 + y=x → y=1
       {A} × {B} = {A,B} → 여전히 차단!
       {C} × {B} = {B,C} → 유효!
       y=1.label = [{B,C}]  ← 우회 성공!
```

**JTMS였다면**: {A,B} 모순 → A 철회 → 모든 추론 재시작 → C 경로 발견

**ATMS는**: {A,B}만 격리하고, {C} 경로를 **자동으로** 찾아냄. 재계산 없음!

## 5. 의의

### 5.1 패러다임 전환 — "참/거짓"에서 "조건부 참"으로

```
JTMS: "X는 참이다"
ATMS: "X는 {A} 하에서 참이고, {B,C} 하에서도 참이다"
```

세계를 하나의 관점이 아닌 **모든 가능한 관점의 집합**으로 표현합니다.

### 5.2 문제 해결자(Problem Solver)와의 분리

```
문제 해결자 → justify-node, assume-node, nogood-nodes 호출
ATMS       → 일관성 유지, label 갱신, 모순 격리 자동 수행
문제 해결자 ← in-node?, interpretations로 결과 조회
```

ATMS는 추론의 논리적 기반만 관리하고, 무엇을 추론할지는 문제 해결자가 결정합니다.

### 5.3 실용적 응용 분야

| 분야 | ATMS 활용 |
|------|-----------|
| **의료 진단** | 여러 질병 가설을 동시에 추적, 검사 결과로 가설 제거 |
| **회로 진단** | 고장 부품 후보를 병렬 추적 (de Kleer의 원래 목적) |
| **설계 탐색** | 여러 설계 대안을 동시에 유지, 제약 위반 시 자동 격리 |
| **계획 수립** | 여러 행동 계획을 병렬 평가, 실패 시 대안 자동 활성화 |

### 5.4 알고리즘적 혁신

- **weave**: 교차곱 + nogood 필터링 + antichain 최소성을 동시에 수행
- **new-nogood 5단계**: 한 번의 호출로 금지 → 정리 → 오염을 일괄 처리
- **환경 테이블의 크기별 정렬**: 부분집합 검사 시 조기 종료 가능

### 5.5 이론적 기여 — Label의 Antichain 불변량

de Kleer가 확립한 핵심 불변량:

> 어떤 노드의 label에 있는 임의의 두 환경 E1, E2에 대해,
> E1 ⊄ E2 이고 E2 ⊄ E1 이다.
> (어느 것도 다른 것의 부분집합이 아니다)

이 덕분에:
- label이 항상 **최소한의 크기**로 유지됨
- "이 노드가 참이 되는 가장 약한 조건들"을 정확히 표현
- 불필요한 중복 없이 모든 가능성을 포괄

### 5.6 저자 de Kleer의 핵심 설계 원칙

1. **발생 시점 처리**: 매번 전수검사 대신, 발생 시 한 번에 전파
2. **최소성 유지**: 테이블에 최소 집합만 보관 → 메모리+속도 최적화
3. **부분집합 판정**: {A,B} ⊆ X 관계로만 오염 → 정확하고 안전
4. **역참조 활용**: env.nodes로 O(k) 빠른 label 정리
5. **이유 기록**: '왜 금지?'를 env에 저장 → 설명 생성 가능

---

## 한 문장 정의

> **ATMS는 "여러 가능한 세계를 동시에 추적하면서, 모순이 발견되면 해당 세계만 격리하고 나머지는 계속 작동하게 하는" 진리 유지 시스템이다.**
